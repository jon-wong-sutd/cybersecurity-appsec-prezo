Bad Access Control Design

Typical Errors

  • Hardcoded Keys
    "My lock works for John Doe and Jane Doe only"
    brittle access control system
  • Security by Obscurity
    "Konami code! up up down down left right a b Enter"
    Eschews keys altogether

Hardcoded keys require the entire locking mechanism to be modified whenever adding access for a new user. This modification can accidentally break the locking mechanism. The proper way to design a locking mechanism is to make sure it is consistently robust, competent, and stable. Hardcoded keys require "breaking open the locking mechanism" just to add new users, making the access control mechanism brittle and unstable.

Security by obscurity eschews keys altogether, requiring specific knowledge to open locks ("secret button(s) behind the piano, under the rug, etc".

Testing Cost per Feature

y-axis: Percentage of feature implementation cost.

x-axis: The n-th feature implemented.

This is one reason why many software developer teams resist testing. TDD incurs a minimum testing cost, which is usually 100% of feature implementation cost for the 1st feature. Traditional testing has a lower starting cost, but ramps up quickly with more interconnected features implemented; developer teams simply cut off testing cost when it raises above 100%.
